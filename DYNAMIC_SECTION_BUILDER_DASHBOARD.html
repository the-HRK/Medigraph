Eliminating Redeploys for Dashboard Changes

StaySync aggregates client intelligence (Financials, Tech Stack, Competitive, etc.). However, client-facing users frequently demand field edits (e.g., adding a table column, changing a KPI). Traditionally, this requires backend schema changes, prompt rewrites, frontend component modifications, and full system redeployments.

The Solution: A dual-LLM pipeline utilizing a Prompt/Schema Refiner and a Production Data Extractor. The system refines prompt texts and lightweight JSON schemas dynamically from natural language input. A **generic dynamic renderer** instantly handles layout adjustments, removing developers entirely from simple field-level change loops.

Architectural Data Pipeline

1

Admin Inputs Request

Admin invokes configuration edit via the UI header. Current config (`promptText` & `schemaDescriptor`) and requested natural language edits are sent to the backend endpoint.

2

Prompt/Schema Refined by LLM Gemini Refiner

A high-speed Gemini model, configured with strict JSON output parameters, reviews the inputs and proposes: an updated `promptText`, an updated output JSON `schemaDescriptor`, and an english `changeSummary`.

3

Visual Diff & Approval Gate

The dashboard administrator receives a clean visual breakdown of changed, added, or removed fields. After human verification, clicking "Approve" triggers backend storage as the active version.

4

Re-Extraction & Storage Production Gemini

The system executes a background call to the heavy extraction model with the new instructions. The structured response JSON is parsed, validated against the base metadata, and stored under `section_data`.

5

Dynamic Renders via WebUI

The StaySync client intelligence frontend fetches the latest dataset plus the active `schemaDescriptor`. It resolves the preset layout template dynamically, using dynamic formatters with zero layout code adjustments.

Database Schema (SQL-Structured)

section_configs Parent Config 
CREATE TABLE section_configs (
  section_id VARCHAR(50) PRIMARY KEY,
  client_id VARCHAR(50) NOT NULL,
  section_type VARCHAR(50) NOT NULL, -- financials, tech-stack, etc.
  layout_template VARCHAR(30) NOT NULL, -- table, kpi-grid, list, etc.
  active_prompt_version_id INTEGER,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
prompt_versions Version History 
CREATE TABLE prompt_versions (
  id SERIAL PRIMARY KEY,
  section_id VARCHAR(50) REFERENCES section_configs(section_id),
  prompt_text TEXT NOT NULL,
  schema_descriptor JSONB NOT NULL, -- keys, types, labels
  change_summary TEXT,
  status VARCHAR(20) DEFAULT 'draft', -- draft, active, superseded
  approved_by VARCHAR(50),
  approved_at TIMESTAMP
);
section_data Extraction Payload 
CREATE TABLE section_data (
  id SERIAL PRIMARY KEY,
  section_id VARCHAR(50) REFERENCES section_configs(section_id),
  prompt_version_id INTEGER REFERENCES prompt_versions(id),
  data JSONB NOT NULL, -- parsed dynamic payload
  fetched_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  validation_status VARCHAR(20) -- success, warning, failed
);

System Risk Assessment & Mitigations

Evaluating the core downsides of dynamic configurations alongside high-quality software engineering guardrails.

LLM Drift & Type Mismatches

The Prompt Refiner generates schema definitions (e.g., expecting a field to be currency), but the Production Extractor might output an unparseable string with nested fields or letters.

StaySync Mitigation

**Defensive Formatting:** The React formatter handles unparseable text gracefully. If a `currency` field receives a text string, the engine outputs the raw string directly instead of throwing an error or displaying `NaN`.

Latency on Extraction

Approving changes triggers an on-demand Production Extraction call, which can cause high page-loading wait states while compiling documents.

StaySync Mitigation

**Draft Queue + Schedulers:** Schema approvals are queued asynchronously. The frontend shows a "Re-indexing data..." loading state and loads cached historical datasets in read-only mode until the background job completes.

Schema Evolution Conflicts

When schemas are dynamically altered, prior executions of `section_data` mismatch the updated active `schemaDescriptor`, breaking history renders.

StaySync Mitigation

**Version Handshaking:** `section_data` binds tightly to `prompt_version_id`. If displaying historical pages, the engine resolves the matching version's schema instead of the current config to render exact historical columns.
